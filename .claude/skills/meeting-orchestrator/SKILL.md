# Skill: meeting-orchestrator
# Goal: Профессиональное ведение встреч Совета с полной транскрипцией и управлением контекстом.

## 🔄 Протокол запуска (ОБЯЗАТЕЛЬНО)
Перед началом любой встречи:
1. **Загрузка памяти**: Найди и прочитай последние 3 лога в `docs/meetings/`.
2. **Анализ состояния**: Сверь текущие файлы проекта с пунктами в `docs/SPECIFICATION.md`.
3. **Reasoning**: Используй тег `<thinking>` для формирования повестки на основе расхождений между ТЗ и кодом.

## 📅 Режим /meeting:daily
Агенты берут инициативу. Формат ответа:
- **@pm**: Открывает встречу. «Сегодня обсуждаем [Screen Name]. Статус ТЗ: [vX.Y].»
- **@product**: Задаёт **ровно один** уточняющий вопрос за раз. Ждёт ответа заказчика.
- **@architect**: Оценивает технические риски предложенных правок (особенно для 5e24).

## 💡 Режим /meeting:idea
- **Фиксация**: @pm записывает идею дословно.
- **Интервью**: @product выявляет «почему» этой идеи через `AskUserQuestion`.
- **Голосование**: Каждый агент обязан выдать вердикт: ✅ / ⚠️ / ❌ с обоснованием.

## 📋 Выходные данные (Success Criteria)
1. Создать файл `docs/meetings/[daily|ideas]/YYYY-MM-DD_[name].md`.
2. **Важно**: В конце лога добавить напоминание: «Используй @spec-maintainer для фиксации решений».
3. **Управление токенами**: Если обсуждение заняло более 15 сообщений, @pm обязан предложить `/clear`.
