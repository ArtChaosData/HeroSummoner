Цель: Гарантировать версионность ТЗ через Git и поэкранную структуру

# Skill: spec-maintainer
# Goal: Управление жизненным циклом ТЗ в docs/SPECIFICATION.md.

## 📐 Структура документа
ТЗ должно быть строго разбито по секциям:
- `# Project Overview`
- `# Screen: [Name]`
  - `## Mechanics (5e14 vs 5e24)`
  - `## UI Requirements`
- `# Change Log`

## 📝 Правила обновления
1. **Atomic Updates**: Обновляй только те экраны, по которым принято решение.
2. **Version Bump**: Увеличивай минорную версию (0.1 -> 0.2) при добавлении новых требований .
3. **Contextual Why**: Каждое изменение должно сопровождаться комментарием в `<think>`, почему это требование важно для баланса.

## 💾 Git Protocol (ОБЯЗАТЕЛЬНО)
После правок выполни: `git commit -m "docs(spec): update [Screen Name] requirements"`.
Верни пользователю вопрос: **«ТЗ закоммичено. Начать реализацию в новой чистой сессии (/clear)?»**.
